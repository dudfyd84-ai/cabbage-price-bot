## 🔗 소유 규칙 — 파일 소유제 ❌ → **기능 담당제** ✅ (2026-07 개정)

처음엔 "C가 화면 파일 전부 소유"로 나눴으나, **화면은 모든 기능이 만나는 접점**이라
실제로는 B(로그인 화면)·D(알림 화면)도 화면을 건드릴 수밖에 없었다. 그래서 규칙을 바꾼다.

**기능 담당자가 그 기능의 화면까지 끝까지 책임진다.**
- B는 로그인·구독 기능이면 `onboarding.html`·`plan.html`까지 직접 수정한다.
- D는 알림 기능이면 `alerts.html`까지 직접 수정한다.
- C는 핵심 화면(홈·재고·BOM·품목상세·딜·주문)과 **공용 모듈**을 담당한다.

### ⚠️ 공용 파일 (여러 기능이 함께 쓰는 것 — 수정 시 절차 필수)
`nav.js` · `ct-store.js` · `home-live.js` · `dashboard.html` · `app.py`

이 파일들을 수정하면.
1. **PR 본문에 "공용 파일 수정: 파일명 — 이유"를 명시**한다.
2. **@kjw0106(C)을 리뷰어로 지정**한다. (공용 모듈 담당)
3. 되도록 **최소 범위**로 고친다. 리팩터링은 별도 PR로.

### 데이터 흐름 접점
- **A → D**: 예측·적중률(A 산출)을 알림·리포트(D)가 소비. 출력 포맷 바뀌면 A가 먼저 공지.
- **B ↔ C**: 새 API는 B가 `app.py`에 추가, C는 호출만. 라우트 명세는 PR 코멘트로 합의.

## 📌 공통 규칙
- 브랜치: `feature/트랙-기능` (예: `feature/A-predict-log`, `feature/C-bom-manual`).
- **한 PR엔 한 이슈만.** 여러 기능이 섞이면 나눠서 올린다(문제 시 되돌리기 쉬움).
- PR 본문에 **`Closes #이슈번호`** 를 반드시 넣는다. 없으면 이슈가 안 닫힌다.
- **작업이 끝나면 push에서 멈추지 말고 PR까지 올린다.** 브랜치에만 있으면 리뷰가 안 간다.
- 리뷰 지적을 반영할 땐 **새 PR을 열지 말고 같은 브랜치에 커밋·push**한다(PR이 자동 갱신됨).
- 데이터·모델 파일(`*.csv`, `*.pkl`, `accuracy.json`)은 재학습 봇이 자동 관리 → 수동 커밋 금지.
- 비밀키는 코드 금지, GitHub Secret만.
